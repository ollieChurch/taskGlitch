# Backlog & Estimation Improvements

**Phase:** 3b | **Version:** 0.19 | **Status:** Not Started

## Problem

The completed tasks list grows indefinitely with no cleanup mechanism, consuming storage and making the list unwieldy. The time estimate evaluation uses exact thresholds without considering that estimates are banded ranges (e.g., a 45-minute task marked "long" is actually correct since the alternative "mid" band would have been further off). Additionally, the blocked tasks feature from v0.17 lacks a dedicated filter, making it harder to find blocked tasks in a large backlog.

## Requirements

### Core

- [ ] Auto-purge completed tasks older than a configurable number of days (default: 90 days)
  - Preserve summary stats (count, total time tracked) for the dashboard before purging
  - User can configure the retention period in their profile/settings
  - Purge runs automatically on login or at a regular interval
- [ ] Fix time estimate evaluation to use banding thresholds:
  - Each size estimate corresponds to a time range (e.g., "Short" = 0-30 min, "Mid" = 30-60 min, "Long" = 60-120 min)
  - A task's actual time should be evaluated against which band it falls into, not an exact threshold
  - The evaluation must use the band values that were in effect at the time of task completion, not the current values
- [ ] Add a filter option for blocked tasks in the backlog filter widget

### Data Model

- [ ] Add to account settings: `completedRetentionDays` (number, default 90)
- [ ] Add to completed task schema: `estimateBandsAtCompletion` (object, nullable) — snapshot of the size band thresholds at the time of completion, for accurate retroactive evaluation
- [ ] Ensure backward compatibility — existing completed tasks without `estimateBandsAtCompletion` fall back to current values

### UX

- [ ] Show a notification or summary when tasks are purged (e.g., "12 completed tasks older than 90 days have been archived")
- [ ] Add retention period setting to the profile/settings page
- [ ] Blocked filter should be available alongside existing priority and size filters
- [ ] Blocked filter should show a count badge indicating how many tasks are currently blocked

### Edge Cases

- [ ] Purge should not delete tasks that have recurrence history links (`recurrenceParentId`) if the parent is still active — or ensure the link is not broken
- [ ] If all completed tasks are purged, dashboard stats should still display correctly from preserved summaries
- [ ] Changing the retention period should not retroactively purge — only apply on next purge cycle
- [ ] Blocked filter combined with other filters should use AND logic (show tasks that are both blocked AND match other filters)

## Files Likely Affected

- `src/composables/useTaskActions.js` — purge logic, time estimate evaluation
- `src/stores/app.js` — account settings, completed task schema
- `src/views/TaskView.vue` — blocked filter integration
- `src/components/FilterWidget.vue` — blocked filter option
- `src/views/ProfileView.vue` or `src/components/SettingsModal.vue` — retention period setting

## Acceptance Criteria

1. [ ] Completed tasks older than the configured retention period are automatically purged
2. [ ] Dashboard stats remain accurate after purging (preserved summaries)
3. [ ] Users can configure the retention period in settings
4. [ ] Time estimate accuracy evaluates against banding thresholds, not exact values
5. [ ] Estimate evaluation uses the band values from the time of completion
6. [ ] Blocked tasks filter is available in the backlog filter widget
7. [ ] Blocked filter shows a count of currently blocked tasks
