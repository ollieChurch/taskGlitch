# Improve Labels Taken From Code

**Phase:** 1 | **Version:** 0.13 | **Status:** Complete

## Problem

Some UI elements display raw values from code or the database rather than human-friendly labels. This makes the app feel unfinished.

## Requirements

### Investigation

- [x] Audit all user-facing text for raw code values. Known issues:
  - Task sizing displays as raw number with "mins" suffix (e.g., "30 mins") — mapped to friendly labels ("Short", "Medium", etc.)
  - Debug `task.score` display gated behind `import.meta.env.DEV`
  - Task type `'userTask'` / `'systemBreak'` used as internal identifiers — confirmed never displayed raw
- [x] Check all console.log statements are removed or gated behind development environment
- [x] Review any tooltips, alt text, or ARIA labels for developer-facing language

### Core

- [x] Create a centralised label mapping for task sizes (15 -> "Short", 30 -> "Medium", 60 -> "Long", 120 -> "Very Long")
- [x] Display friendly labels in task cards, modals, and schedule views
- [x] ~~Show the minute value as secondary info~~ **Decision: label only, no minutes** — t-shirt sizing philosophy means users think in relative terms, not minutes
- [x] Ensure the debug score display is gated behind a development-only flag
- [x] Remove or gate all console.log statements behind environment checks

### Consistency

- [x] All priority labels capitalised from store definitions (Critical, High, Medium, Low)
- [x] Category names remain user-entered (no forced title-casing)
- [x] Date formats consistent across the app (en-uk locale)

## Implementation Notes

- Added `sizeLabels` map and `getSizeLabel` getter to Pinia store
